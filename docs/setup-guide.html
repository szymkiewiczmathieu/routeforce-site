RouteForce Setup Guide

Version: March 2026
Audience: Salesforce administrators during initial deployment
Goal: Install RouteForce, assign access, configure core settings, and validate the first working rollout

1. Before you start

Prerequisites

  • Salesforce org with Lightning enabled
  • System Administrator access
  • RouteForce package installation link
  • RouteForce licence key
  • ORS API key if provided for your environment
  • Geocoded data on the records you want to display

Recommended preparation

  • Confirm whether your org should use Shipping or Billing addresses.
  • Identify pilot users.
  • Check that your key Account records already contain latitude and longitude.

2. Install the package

Use the RouteForce installation URL provided by your RouteForce contact.

Environment Typical endpoint
Production https://login.salesforce.com/packaging/installPackage.apexp?...
Sandbox https://test.salesforce.com/packaging/installPackage.apexp?...

Installation recommendation

  • Choose Install for Admins Only first.
  • Assign users intentionally after validation.

During installation

Approve access requests if Salesforce prompts for RouteForce callout endpoints.

3. Assign permission sets

RouteForce ships with two main permission sets.

Permission Set Intended audience Purpose
RouteForce Admin Salesforce admins Full setup and configuration access
RouteForce User Sales reps and field users Daily operational access

Assign in Salesforce

  1. Go to Setup > Permission Sets.
  2. Open the permission set.
  3. Click Manage Assignments.
  4. Add users.

Optional CLI assignment

sf org assign permset --name RouteForce_Admin --target-org <your-org>
sf org assign permset --name RouteForce_User --target-org <your-org>

4. Launch the onboarding wizard

Open the RouteForce app and go to the Configuration tab.

Run the RF_Onboarding flow.

Step 1 — API keys

Provide:

  • Licence Key
  • ORS API Key
  • Fallback API Key if used in your deployment

Step 2 — map defaults

Set:

  • address prefix
  • default map layer
  • default zoom
  • default display mode

Step 3 — feature toggles

Enable or disable features such as:

  • Events
  • Leads
  • Nearby
  • Visit Reports
  • History naming
  • Opportunities

Step 4 — advanced settings

Typical advanced settings:

  • cluster radius
  • max route steps
  • nearby radius
  • support email

5. Configure the default user experience

Use the onboarding flow and Lightning App Builder together.

Good default choices

Setting Recommended starting point
Address prefix Shipping if visits happen at shipping sites
Map layer Standard
Display mode Accounts first
Leads Disabled unless field teams use them daily
Opportunities Enable only if route planning needs them
Nearby radius Conservative to avoid clutter

FlexiPage customization

In Lightning App Builder, you can configure:

  • filter fields
  • legend values
  • marker popup fields
  • record type filter
  • multi-select behavior

6. Validate route settings

Make sure the route panel behaves as expected.

Check that users can:

  • open route mode
  • choose a starting point
  • add stops
  • edit route timing
  • optimize successfully

Review these core settings in particular:

  • departure time
  • lunch break
  • end of day
  • visit duration
  • max route steps

7. Verify remote access

RouteForce relies on external routing and licence services.

Verify Remote Site Settings if needed

Typical endpoints include:

  • https://ors.routeforce.app
  • https://api.openrouteservice.org

If they are not already available through the package, add them manually in Setup > Remote Site Settings.

8. Verify your geocoded data

If the data is not geocoded, users will see an empty map.

Accounts

Use either:

  • ShippingLatitude / ShippingLongitude
  • or BillingLatitude / BillingLongitude

Leads

Use standard Lead latitude / longitude fields.

Opportunities

Opportunities use the parent Account address.

Quick SOQL check

SELECT Id, Name, ShippingLatitude, ShippingLongitude
FROM Account
WHERE ShippingLatitude != null
LIMIT 20

If you use Billing instead:

SELECT Id, Name, BillingLatitude, BillingLongitude
FROM Account
WHERE BillingLatitude != null
LIMIT 20

9. Run a full installation test

Use a small pilot scenario before broad rollout.

Validation checklist

  1. Open the RouteForce tab.
  2. Confirm markers are visible.
  3. Apply at least one filter.
  4. Add 2 to 5 stops.
  5. Optimize the route.
  6. Confirm distance and timing are calculated.
  7. Create route visits if relevant.
  8. Check route history.

If this works cleanly, the setup is ready for pilot users.

10. Recommended rollout order

Phase 1 — Admin validation

  • install package
  • configure defaults
  • confirm licence
  • validate routing

Phase 2 — Pilot users

  • assign RouteForce User to a small group
  • gather feedback on filters and marker fields
  • adjust defaults if needed

Phase 3 — Broader deployment

  • assign more users
  • train on route creation and check-in
  • publish user guide internally

11. Common setup issues

Issue Likely cause Fix
Map is empty No geocoded records Verify latitude / longitude fields
Licence error Missing or invalid licence key Re-run onboarding and update keys
Route optimization fails ORS access issue Verify keys and remote access
Users cannot open RouteForce Missing permission set Assign RouteForce User
Wrong addresses used Wrong address prefix Switch Shipping vs Billing
Too much clutter on map Too many objects enabled by default Simplify display mode and filters

12. Handover to operations

Before closing the setup phase, make sure you have:

  • documented the chosen address model
  • documented enabled features
  • confirmed support email
  • identified the admin owner of RouteForce
  • shared the user guide with field teams

Support

For package access, licence activation, or routing issues, use your RouteForce support channel and keep your org name plus environment details handy.
