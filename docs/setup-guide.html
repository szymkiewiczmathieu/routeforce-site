Setup Guide

Install RouteForce, assign access, configure core settings, and validate your first rollout.

Prerequisites

  • Salesforce org (Enterprise, Unlimited, or Developer Edition)
  • System Administrator profile
  • Translation Workbench enabled (Setup > Translation Workbench > Enable)
  • Languages activated: English, French, Spanish (if multilingual)
  • Optional: Account Teams enabled (Setup > Account Teams Settings) — required for the "My Account Team" filter
  • Optional: Team Selling enabled (Setup > Opportunity Team Settings) — required for the "My Opportunity Team" filter

1. Install the Package

Install the RouteForce managed package using the installation URL provided by your administrator.

Environment Link
Production Install v5.1.0
Sandbox Install v5.1.0

Choose Install for Admins Only (recommended), then assign permission sets manually.

2. Assign Permission Sets

RouteForce ships two permission sets:

Permission Set Target Users Capabilities
RouteForce Admin Salesforce Admins Full access + Configuration tab + Licence tab + modifyAllRecords
RouteForce User Sales Reps Standard access to map, routes, history, filters, visit reports

Assign via Setup > Permission Sets or:

sf org assign permset --name RouteForce_Admin --target-org <your-org>
sf org assign permset --name RouteForce_User --target-org <your-org>

Enable "Run Flows" Permission (Required)

RouteForce uses Screen Flows (e.g., Create Event, Custom Actions). The Standard User profile does not have the "Run Flows" system permission enabled by default. Without it, users will see:

"You do not have the level of access necessary to perform the operation you requested."

This permission cannot be granted via a managed package — it must be configured manually by the admin:

Option A — Enable on the Profile:
1. Go to Setup > Profiles > Standard User (or the relevant profile)
2. Under System Permissions, enable Run Flows
3. Save

Option B — Create a Permission Set:
1. Go to Setup > Permission Sets > New
2. Name it (e.g., "Run Flows Access")
3. Under System Permissions, enable Run Flows
4. Assign it to all RouteForce users

3. Assign the RouteForce App

After assigning permission sets, make the RouteForce app visible to users:

  1. Go to Setup > App Manager
  2. Find RouteForce in the list
  3. Click the dropdown arrow > Manage
  4. Under Profiles or Permission Sets, assign the app to the relevant profiles/permission sets

Alternatively, the RouteForce_Admin and RouteForce_User permission sets already include app visibility — users with either permission set will see RouteForce in the App Launcher.

Field-Level Security Notes

RouteForce uses WITH USER_MODE for all SOQL queries, which enforces Field-Level Security (FLS). The included permission sets grant the necessary FLS, but be aware of the following requirements:

Requirement Fields Needed
Accounts Read access to address fields (Shipping or Billing, depending on AddressFieldPrefix__c)
Events Read/Edit access to Subject, StartDateTime, EndDateTime, Description, WhoId, WhatId
Leads (if enabled) Read access to Lead address fields (Address, Latitude, Longitude)
Opportunities (if enabled) Read access to Opportunity fields + parent Account address

The RouteForce_User permission set grants all required FLS. If you use custom profiles with restricted FLS, ensure the above fields are accessible.

4. Run the Onboarding Wizard

  1. Open the RouteForce app from the App Launcher
  2. Navigate to the Configuration tab
  3. Launch the RF_Onboarding screen flow
  4. Enter your API keys:
Field Required Description
Licence Key Yes Your RouteForce licence key (provided by RouteForce support)
ORS API Key Yes API key for the VPS routing server
Fallback API Key No HeiGIT fallback key (used when VPS is unavailable)

Keys are stored securely in a protected Custom Setting (InternalConfig__c), invisible to standard users.

After saving the keys, the wizard directs you to configure application settings via Custom Metadata Types:

Setup → Custom Metadata Types → RouteForce Config → Manage Records → Default

See the Configuration Guide for all available settings (map defaults, feature toggles, route settings, query filters, etc.).

Visit Report Settings (Recommended)

Two optional settings enhance the visit report experience:

CMDT Field Where Description
VisitAlertDays__c Setup > Custom Metadata > RouteForce Config > Default Number of days after which a record is flagged as overdue. Markers display a red alert badge, and Nearby suggestions prioritize overdue records. Set to 0 to disable (default)
VisitResultOptions__c Setup > Custom Metadata > RouteForce Config > Default Comma-separated picklist values for the visit result field (e.g., Successful,Follow-up needed,No contact,Cancelled). Leave blank for free text input

7. Custom Action Buttons (Optional)

Add custom buttons to marker popups that launch your own Screen Flows (e.g., data entry forms, checklists).

Step 1 — Create a Screen Flow

  1. Create a Screen Flow with a recordId Text input variable (mark as "Available for Input")
  2. Add a single Screen element containing your LWC component
  3. Pass {!recordId} to the component's recordId attribute
  4. Activate the flow

Step 2 — Register the Action

  1. Go to Setup > Custom Metadata Types > RouteForce Action > Manage Records
  2. Click New
  3. Fill in:
  4. Label: Button text (e.g., "Data Entry")
  5. Flow API Name: Your flow API name (e.g., RF_SaisieGmsDN)
  6. Object Type: Which markers show this button (All, Account, Lead, Opportunity)
  7. Display Context: Where the button appears (All, Popup, or EventPanel)
  8. Button Color: Hex color (e.g., #F59E0B)
  9. Icon: action, edit, check, chart, list, or star
  10. Filter Condition (optional): Show this button only when a record field matches a value. Syntax: FieldName=Value (equals), FieldName!=Value (not equals), FieldName=Val1,Val2 (in list). The field must be included in the FlexiPage's markerFields or be a standard marker property.
  11. Save — the button appears immediately in marker popups

8. Event Layout — Visit Report Field (Recommended)

RouteForce adds a custom lookup field Visit Report (VisitReport__c) on the standard Activity object. This field links each Event to the Visit Report created during check-in, allowing users to navigate directly from an Event to its associated report.

FLS is already granted by both RouteForce_Admin and RouteForce_User permission sets — no additional configuration is needed for field access.

To make this field visible on the Event record page:

  1. Go to Setup > Object Manager > Event > Page Layouts
  2. Edit your Event page layout
  3. Drag the Visit Report field from the palette onto the layout (e.g., in the "Information" section)
  4. Save the layout

Note: This step is optional but recommended. Without it, users won't see the link to the Visit Report from their Event records.

  • [ ] Map loads — Map displays with geocoded Account markers

  • [ ] Events panel — Today's events load (if EnableEvents__c = true)
  • [ ] Route optimization — Create a route with 3+ stops and optimize it
  • [ ] Check-in works — Check in on an event and verify Visit Report is created (if EnableEvents__c = true)
  • [ ] Licence valid — Check the Licence tab — should show "Active"
  • [ ] Leads visible — Lead markers appear on map (if EnableLeads__c = true)
  • [ ] Opportunities visible — Opportunity markers appear (if EnableOpportunities__c = true)
  • [ ] Background jobs scheduled — Verify the following 3 jobs appear in Setup > Scheduled Jobs (auto-configured since v5.0.0):
  • RouteForce Usage Aggregation — daily at 1:00 AM
  • RouteForce Log Purge — daily at 2:00 AM
  • RouteForce Usage Purge — weekly on Sunday at 3:00 AM

10. Background Jobs (Auto-Configured) — New in v5.0.0

Starting with v5.0.0, RouteForce includes a PostInstallHandler that automatically schedules all background jobs when the package is installed or upgraded. No manual System.schedule() calls are needed.

Jobs Scheduled Automatically

Job Name Schedule Purpose
RouteForce Usage Aggregation Daily at 1:00 AM Rolls up usage detail records into daily summaries
RouteForce Log Purge Daily at 2:00 AM Deletes logs older than 30 days
RouteForce Usage Purge Weekly on Sunday at 3:00 AM Purges old usage summary records

Self-Rescheduling

Each job automatically re-schedules itself after every run. This means the jobs survive scenarios that would normally break standard Salesforce scheduled jobs (e.g., the scheduling user being deactivated). The job simply re-creates its schedule under the running user context.

Reschedule Button (Troubleshooting)

If background jobs stop running (e.g., after a sandbox refresh, which clears all scheduled jobs), admins can restore them without writing any code:

  1. Open the RouteForce app
  2. Navigate to the Usage Dashboard tab
  3. Click the Reschedule Jobs button

This aborts any stale entries and re-schedules all three jobs with their default cron expressions.

Post-Install Verification

After installing or upgrading to v5.0.0+, confirm the jobs are active:

  1. Go to Setup > Scheduled Jobs
  2. Verify all three jobs are listed with a WAITING state:
  3. RouteForce Usage Aggregation
  4. RouteForce Log Purge
  5. RouteForce Usage Purge

If any job is missing, use the Reschedule Jobs button in the Usage Dashboard (see above).

11. Logging & Log Purge

RouteForce includes a built-in structured logging system that records application events in the RouteForceLog__c custom object. Logs are created automatically by the application across all services (map, routing, events, filters, configuration, licence) — no manual setup is required.

What Gets Logged

Level Examples
INFO Batch completion, configuration changes, route saves
WARN API fallback triggered, unexpected data format
ERROR Callout failures, DML errors, licence validation failures

Viewing Logs

  1. Navigate to the RouteForce Logs tab (visible to Admins via RouteForce_Admin permission set; hidden for standard users by default)
  2. Use list views or reports to filter by Level, Context, User, or date range
  3. Each log record shows: Level, Context (which service), Message, Timestamp, and User

Automatic Purge

Logs older than 30 days are automatically deleted by the LogPurgeBatch scheduled job. As of v5.0.0, this job is scheduled automatically on install — no manual setup is needed (see Background Jobs).

Custom Retention Period

The default retention is 30 days. To use a different period, schedule manually with a custom value via Anonymous Apex:

// Keep logs for 90 days instead of 30
System.schedule('RouteForce Log Purge', '0 0 2 * * ?', new routeforce.LogPurgeSchedulable(90));

Note: If you manually schedule with a custom retention, the PostInstallHandler will not override it on future upgrades as long as a job with the same name is already active.

Storage Considerations

Each log record consumes standard data storage. For high-volume orgs, consider:
- Reducing retention to 7–14 days (see Custom Retention Period above)
- Monitoring data storage via Setup > Storage Usage

12. Usage Tracking & Analytics (Optional)

RouteForce includes an opt-in usage tracking system that records user actions (route creation, optimization, check-ins, event creation, filters, exports, address enrichment) for analytics purposes.

How It Works

  1. Real-time tracking — Each user action creates a detail record in RouteForceUsage__c (async via @future to avoid impacting the main transaction)
  2. Daily aggregation — The UsageAggregationBatch rolls up detail records into daily summaries per user/action in RouteForceUsageSummary__c, then deletes processed details to save storage
  3. Dashboard — The Usage Dashboard tab reads from summaries for fast reporting

Enable Usage Tracking

Usage tracking is disabled by default. To enable it:

  1. Go to Setup > Custom Metadata Types > RouteForce Config > Manage Records > Default
  2. Set EnableUsageTracking__c to true
  3. Save

Aggregation Batch (Auto-Scheduled)

As of v5.0.0, the aggregation batch is scheduled automatically on install — no manual setup is needed (see Background Jobs). The batch runs daily at 1:00 AM and aggregates the previous day's detail records into summaries.

Actions Tracked

Action Description
Route Created User saves a new route
Route Optimized User optimizes a route
Check-In User checks in on a visit
Event Created User creates calendar events from a route
Filter Applied User applies map filters
Address Enriched User enriches account addresses
CSV Exported User exports data as CSV
PDF Exported User exports data as PDF

Storage Considerations

Detail records are automatically deleted after aggregation. Only summary records are retained long-term (one record per user/action/day). For most orgs, storage impact is minimal.

Multilingual Support

RouteForce supports English, French, and Spanish out of the box.

  • All Custom Labels are translated
  • Object labels and tab names are translated via objectTranslations
  • The UI language follows the user's Salesforce language preference

To verify: change your language in Settings > Language & Time Zone and refresh the app.

Troubleshooting

Issue Solution
Map doesn't load Check Remote Site Settings and licence key
"Licence not configured" Run the RF_Onboarding flow and enter your licence key
No markers on map Ensure Accounts have geocoded addresses (ShippingLatitude/Longitude or BillingLatitude/Longitude)
Route optimization fails Check that ORS API Key is configured and VPS is reachable
Tabs not translated Enable Translation Workbench and activate FR/ES languages
