Configuration Guide

A complete reference to configure, customize, and maintain RouteForce in your Salesforce org.

A complete reference for Salesforce administrators to configure, customize, and maintain RouteForce.

1. Configuration Architecture

RouteForce uses a CMDT-first approach. All configurable settings are stored in Custom Metadata Types, which are:

  • Deployable across environments
  • Editable without code changes
  • Accessible to the onboarding wizard (Screen Flow)
RF_Onboarding Flow (Screen Flow)
    │
    └─► SecureConfigService ──► InternalConfig__c (API keys, protected)

RouteForceConfig__mdt (app settings, edited directly in Setup)
    │
    ▼
ConfigService.cls (Apex reader)
    │
    ▼
LWC Components (@wire)

Team Filter — Prerequisites

When EnableTeamFilter__c is enabled, ownership filter comboboxes gain an additional option: "My Account Team" (in the Account section) and "My Opportunity Team" (in the Opportunity section). This shows records where the current user is a member of the Account Team or Opportunity Team.

Requirements:

  • Account Teams must be enabled: Setup > Account Teams Settings > Enable Account Teams
  • Team Selling must be enabled (for Opportunities): Setup > Opportunity Team Settings > Enable Team Selling
  • If these features are not enabled in the subscriber org, the filter falls back gracefully to "My Accounts" / "My Opportunities" (owner filter)
  • Leads do not have a native Team concept — the Lead ownership filter only offers "All Leads" and "My Leads"

Role Filter — How It Works

When EnableRoleFilter__c is enabled, ownership filter comboboxes gain a "My Role Team" option for all three objects (Accounts, Leads, Opportunities). This shows records owned by any active user whose role is the same as or subordinate to the current user's role in the role hierarchy.

Key details:

  • Uses 2 SOQL queries: one to load all UserRoles, one to get active users in matching roles
  • The hierarchy is traversed in memory — works with any depth of nesting
  • If the current user has no role assigned, "My Role Team" returns only their own records
  • No prerequisites — unlike Team Filter, this works out of the box in any Salesforce org with a role hierarchy
  • Both EnableTeamFilter__c and EnableRoleFilter__c can be enabled simultaneously — they appear as separate options in each combobox

Independent Ownership Filters

Each object type (Account, Lead, Opportunity) has its own independent ownership filter in its respective filter section. This means you can view "My Accounts" while simultaneously viewing "All Leads" and "My Opportunities".

Object Filter field Options available
Account onlyMyAccounts All / My / My Account Team / My Role Team
Lead onlyMyLeads All / My / My Role Team*
Opportunity onlyMyOpportunities All / My / My Opportunity Team / My Role Team

* Requires the corresponding feature toggle (EnableTeamFilter__c or EnableRoleFilter__c) to be enabled.

Saved filters store each ownership setting independently. Filters saved before this feature used a single onlyMyAccounts value for all objects — these are automatically migrated: the saved value is applied to all three ownership filters when loaded.

Query Filter Security

Admin-provided SOQL filters (ContactQueryFilter__c, LeadQueryFilter__c, OpportunityQueryFilter__c) are validated using an allowlist regex before being injected into queries. Only well-formed WHERE clause predicates are accepted:

  • Allowed: field comparisons (=, !=, <, >, <=, >=), LIKE, IN, NOT IN, INCLUDES, EXCLUDES
  • Allowed values: strings ('...'), numbers, booleans (TRUE/FALSE), NULL, date literals (LAST_N_DAYS:30, TODAY, THIS_MONTH, etc.)
  • Allowed operators: AND, OR, parentheses
  • Rejected: anything that doesn't match the pattern (e.g., subqueries, DML keywords, semicolons)

If a filter is rejected, the system falls back to the default filter and logs a warning in RouteForceLog__c.

3. API Key Management

API keys are stored in InternalConfig__c (Hierarchy Custom Setting), protected by the routeforce namespace — subscribers cannot read them.

Key Purpose Stored In
Licence Key RouteForce licence validation InternalConfig__c.LicenceKey__c
ORS API Key VPS routing server (ors.routeforce.app) InternalConfig__c.OrsApiKey__c
Fallback API Key HeiGIT public API (fallback routing) InternalConfig__c.FallbackApiKey__c

These keys are set during the RF_Onboarding wizard (Step 1 — API Keys) and stored via SecureConfigService.storeKeys().

4. Licence System

How It Works

  1. Validation callout → https://ors.routeforce.app/api/licence/{key}
  2. Server returns: { valid: true, expires: "2026-12-31", client: "Acme Corp" }
  3. Result is cached in LicenceCache__c for 24 hours
  4. If the server is unreachable, the app continues for 7 days (fail-open grace period)
  5. After 7 days without contact, the app blocks with a licence error overlay

Licence States

State Behavior
Active Normal operation. Green indicator in Configuration section
Inactive Licence expired or invalid. Red indicator, app blocked
Not Configured No licence key entered. Setup prompt displayed
Server Unreachable Fail-open: app works for 7 days. Warning indicator

Cache Details

Field Description
LicenceCache__c.IsValid__c true if last validation succeeded
LicenceCache__c.CheckedAt__c Timestamp of last callout
LicenceCache__c.ClientName__c Client name from server response
LicenceCache__c.ExpiresAt__c Licence expiration date

HTTP Response Handling

Code Meaning Action
200 Success Update cache, allow/block based on valid field
401 Invalid key Block immediately (no fail-open)
403 Kill switch Block immediately (nginx-level block)
5xx Server error Fail-open for 7 days
Timeout Unreachable Fail-open for 7 days

5. FlexiPage Configuration

The RouteForce FlexiPage is the main map interface. Configure it via Setup > Lightning App Builder > RouteForce.

Filter Fields

Define which picklist/boolean fields appear in the filter panel using a single comma-separated filterFields property. Each field must be prefixed with its object name.

Syntax Example Description
Object-prefixed Account.Industry Account field
Object-prefixed Opportunity.StageName Opportunity field
Object-prefixed Lead.Status Lead field
Cross-object Opportunity.Account.Industry Filter Opportunities by their Account's field value

Supported field types: Picklist, Multi-Picklist, Boolean (Checkbox)

CMDT Fallback: If the FlexiPage filterFields property is left empty, the system reads the default from RouteForceConfig__mdt.FilterFields__c. The FlexiPage value always takes priority over the CMDT default.

7. Data Requirements

Account Geocoding

Accounts must have latitude/longitude fields populated to appear on the map:

Address Prefix Latitude Field Longitude Field
Shipping ShippingLatitude ShippingLongitude
Billing BillingLatitude BillingLongitude

How to geocode: Use third-party geocoding tools (Geocodio, Google Geocoding API, Salesforce Data Cloud) to batch-update latitude/longitude coordinates on your records. You can also use Data Loader or an ETL tool to import pre-geocoded data.

Tip: Records without coordinates are silently excluded from the map — no error is thrown.

Lead Geocoding

Leads use their standard Latitude and Longitude fields. These are part of the Lead Address compound field.

Opportunity Geocoding

Opportunities do not have their own address. RouteForce uses the parent Account's address (via the AccountId relationship). The Account must be geocoded.

Events

Events appear on the map when they have a related record (WhatId or WhoId) that has geocoded coordinates. Events without a geocoded related record show in the calendar but not on the map.

Filter Condition Syntax

The FilterCondition__c field allows you to conditionally display an action button based on a field value on the record. The field must be included in the FlexiPage's markerFields property or be a standard marker property (type, city, name, etc.).

Syntax Example Description
Field=Value Type=Customer - Direct Show only when field equals value (case-insensitive)
Field!=Value Type!=Prospect Show only when field does NOT equal value
Field=Val1,Val2 Type=Customer - Direct,Customer - Channel Show when field matches any value in comma-separated list

Examples:

# Only show on Account Type "Distributeur - Utilisateur Actif"
Type=Distributeur - Utilisateur Actif

# Show on multiple types
Type=Customer - Direct,Customer - Channel

# Exclude Prospects
Type!=Prospect

# Filter on a custom field (must be in markerFields)
Enseigne_utilisateur__c=Carrefour,Leclerc

How It Works

  1. The admin creates a RouteForceAction__mdt record (Setup > Custom Metadata Types > RouteForce Action > Manage Records)
  2. The admin creates a Screen Flow that accepts a recordId Text input variable
  3. The Screen Flow contains a single screen with the desired LWC component (e.g., a data entry form)
  4. When a user clicks the custom button in a marker popup, the flow opens in a fullscreen responsive modal
  5. The flow receives the record ID of the clicked marker

Important: The Screen Flow must define a Text input variable named recordId which receives the Salesforce ID of the clicked record (Account, Lead, or Opportunity).

Example: Custom Action with Filter Condition

Label:           DN Entry
FlowApiName:     RF_SaisieGmsDN
ObjectType:      Account
ButtonColor:     #F59E0B
Icon:            edit
SortOrder:       1
IsActive:        true
DisplayContext:  All
FilterCondition: Type=Distributeur - Utilisateur Actif

This button only appears on Account markers where Type equals "Distributeur - Utilisateur Actif".

11. Adding Custom Filter Fields

Example: Add "Account.NumberOfEmployees" as a filter

  1. Open Setup > Lightning App Builder
  2. Edit the RouteForce AppPage
  3. Click the RouteForce component
  4. Find the Filter fields property
  5. Append ,Account.NumberOfEmployees to the existing CSV
  6. Save and activate

Note: Only Picklist, Multi-Picklist, and Boolean fields are supported as filters. Text and Number fields are not filterable (but can be shown in marker popups via markerFields).

Example: Add a cross-object filter

To filter Opportunities by their parent Account's Rating:

  1. Find the Filter fields property
  2. Append ,Opportunity.Account.Rating to the existing CSV
  3. Save

This adds a Rating filter that, when selected, only shows Opportunities whose parent Account matches the selected Rating value.

Example: Add a field to marker popups

To show Opportunity Probability in popups:

  1. Find the markerFields property
  2. Add Opportunity.Probability to the comma-separated list
  3. Save

13. Multi-Select Filter Configuration

By default, picklist filters render as single-select dropdowns. To render them as multi-select checkbox lists (allowing multiple selections):

14. Record Type Filter

Record Type filters are auto-detected: if an object (Account, Lead, or Opportunity) has more than one active Record Type, a filter dropdown appears automatically in that object's filter section. No configuration needed.

15. "Not Visited Since" Filter

The "Not visited since" filter is a shared filter displayed in the Account filter section. It applies simultaneously to all 3 object types (Account, Lead, Opportunity).

Data source: The last visit date is automatically derived from the most recent VisitReport__c record linked to each Account, Lead, or Opportunity. No admin configuration is needed — the filter and popup date appear automatically when Visit Reports are enabled (EnableVisitReports__c = true, which is the default).

The NotVisitedSinceDays__c threshold (default: 30 days) is shared across all objects.

Filter options: All, X days (dynamic from CMDT), 3 months, 6 months, 12 months, 24 months, Never visited.

16. Routing Architecture

Primary: VPS ORS

  • URL: https://ors.routeforce.app
  • Auth: API key in X-API-Key header
  • Services: VROOM optimization (optimal stop order) + Directions (route geometry)
  • Limit: 50 waypoints per request

Routing Profile & Speed Estimates

RouteForce uses the driving-car profile (hardcoded). Speed estimates are derived from OpenStreetMap road data:

Road Type Estimated Speed
Highway / Motorway ~120–130 km/h
National / Trunk Road ~80–90 km/h
Urban Road ~40–50 km/h
Residential Street ~20–30 km/h

These speeds are ORS defaults — they cannot be configured per-customer today. Travel time estimates shown in the route panel are calculated using these speeds combined with actual road network distance.

Fallback: HeiGIT Public API

  • URL: https://api.openrouteservice.org
  • Auth: API key in Authorization header
  • Used when: VPS returns 5xx, times out, or throws CalloutException
  • Limit: Lower rate limits (public tier)

Kill Switch

The VPS nginx can disable the X-API-Key globally, returning 401 for all requests. This blocks both routing and licence validation simultaneously.

Error Handling

Error User Impact
400 (Bad Request) "Invalid coordinates" — point is not routable (ocean, no road data)
401 (Unauthorized) "API key invalid" — check InternalConfig__c keys
5xx (Server Error) Automatic fallback to HeiGIT. If both fail, error toast
Timeout Automatic retry (3 attempts), then fallback

20. Logging & Monitoring

Architecture

RouteForce uses a buffer-and-flush logging pattern:

Apex Service (e.g., RouteService)
    │
    ├─► Logger.info('RouteService', 'Optimization completed')
    ├─► Logger.warn('RouteService', 'Fallback to HeiGIT')
    └─► Logger.error('RouteService', 'Callout failed: ' + e.getMessage())
          │
          ▼
    Logger.flush()  →  INSERT RouteForceLog__c[]
  • Log entries are buffered in memory during the transaction
  • Logger.flush() writes all buffered entries in a single DML statement
  • Flush runs in system mode — logs are always written regardless of user permissions
  • If flush fails (DML exception), the error is logged to System.debug and the calling transaction is not disrupted

21. Background Jobs & Automation

Overview

RouteForce runs 3 scheduled background jobs for data maintenance and analytics. These jobs are automatically scheduled on package install or upgrade — no manual setup is required.

PostInstallHandler — Automatic Scheduling

When RouteForce is installed or upgraded, the PostInstallHandler class automatically schedules all 3 background jobs:

Job Name Class Schedule Description
RouteForce Usage Aggregation UsageAggregationSchedulable Daily at 1:00 AM Aggregates raw usage tracking records into daily/weekly summaries
RouteForce Log Purge LogPurgeSchedulable Daily at 2:00 AM Deletes RouteForceLog__c records older than 30 days
RouteForce Usage Purge UsagePurgeBatch Weekly, Sunday 3:00 AM Deletes raw usage tracking records older than 90 days

The handler checks for existing scheduled jobs before creating new ones, so upgrading the package does not create duplicate jobs.

Self-Healing Jobs (Self-Rescheduling Pattern)

Each batch job implements a self-rescheduling pattern for resilience:

  1. When a batch's finish() method runs, it aborts its own CronTrigger and immediately re-schedules itself
  2. This means the job always runs under the context of the user who last triggered it
  3. If the original scheduling user is deactivated (e.g., an admin leaves the company), the job still executes because it re-schedules under the current execution context

This pattern ensures jobs never silently stop due to user deactivation — a common issue with standard Salesforce scheduled jobs.

Job Status — Usage Dashboard

The Usage Dashboard (accessible to admins) displays a Job Status table showing the health of all 3 background jobs:

Column Description
Job Name Name of the scheduled job
Status Color-coded badge: green = Active, orange = Inactive, gray = Missing
Scheduled By Name of the user who last scheduled the job
Next Run Next scheduled fire time
  • Active — the job is scheduled and will run at the next fire time
  • Inactive — a CronTrigger exists but is not in a running state
  • Missing — no CronTrigger found for this job (needs rescheduling)

Reschedule Jobs Button

If jobs are missing or inactive (e.g., after a sandbox refresh, or if an admin manually deleted them), the Usage Dashboard provides a Reschedule Jobs button:

  • Visibility: only shown to users with the RouteForce_Admin permission set
  • Action: aborts all 3 existing CronTriggers (if any) and re-schedules them under the current admin user
  • Result: all 3 jobs return to Active status with updated "Scheduled By" and "Next Run" values

This is a one-click recovery mechanism — no need to run anonymous Apex or navigate to Setup.

Manual Scheduling (Advanced)

In rare cases where you need to schedule jobs manually (e.g., from Developer Console), use:

// Usage Aggregation — daily at 1 AM
System.schedule('RouteForce Usage Aggregation', '0 0 1 * * ?', new routeforce.UsageAggregationSchedulable());

// Log Purge — daily at 2 AM
System.schedule('RouteForce Log Purge', '0 0 2 * * ?', new routeforce.LogPurgeSchedulable());

// Usage Purge — weekly Sunday at 3 AM
System.schedule('RouteForce Usage Purge', '0 0 3 ? * SUN', new routeforce.UsagePurgeBatch());
